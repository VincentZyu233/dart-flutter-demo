# Windows Runner Plan

## Goal

Replace the current "generate Windows project in CI, then patch it" flow with a checked-in `windows/` runner so the FFI integration is stable and reviewable.

## Current Problem

The repository does not contain a persistent `windows/` directory. GitHub Actions runs `flutter create --platforms=windows` and then patches `windows/runner/CMakeLists.txt` with string replacements. This is fragile because it depends on the generated template staying structurally identical across Flutter versions.

## Minimal Checked-In Files

To stabilize the Windows build, check in the full `windows/` directory generated by Flutter for this app, including:

- `windows/CMakeLists.txt`
- `windows/flutter/CMakeLists.txt`
- `windows/flutter/generated_plugins.cmake`
- `windows/runner/CMakeLists.txt`
- `windows/runner/main.cpp`
- `windows/runner/flutter_window.cpp`
- `windows/runner/flutter_window.h`
- `windows/runner/utils.cpp`
- `windows/runner/utils.h`
- `windows/runner/win32_window.cpp`
- `windows/runner/win32_window.h`
- `windows/runner/Runner.rc`
- `windows/runner/resource.h`
- `windows/runner/resources/`

The generated `windows/flutter/generated_plugins.cmake` may still be refreshed by Flutter tooling, but the runner layout and CMake wiring should live in the repository.

## Required Permanent Customization

After checking in the runner, make the following permanent edits once:

1. Add `runner/plugins/system_info/system_info_plugin.cpp` to `windows/runner/CMakeLists.txt`
2. Add `runner/plugins/system_info` to the runner include directories
3. Add `/EXPORT:GetSystemInfoJson` and `/EXPORT:FreeSystemInfoJson` to the runner link options
4. Copy `plugins/windows/SystemInfo.ps1` into the packaged runtime directory as part of the Windows build or install step
5. Keep `plugins/windows/system_info_plugin.h` and `.cpp` as the source of truth, then either:
   - copy them into `windows/runner/plugins/system_info/` when updating the runner, or
   - move the source of truth directly under `windows/runner/plugins/system_info/`

## CI Simplification After Check-In

Once `windows/` is committed, the Windows part of CI can be simplified to:

1. `flutter pub get`
2. `flutter build windows --release`
3. copy `plugins/windows/SystemInfo.ps1` into `build/windows/x64/runner/Release/plugins/windows/`

That means these CI steps can be removed for Windows:

- `flutter create --platforms=windows`
- string patching of `windows/runner/CMakeLists.txt`
- temporary copy into a generated `windows/runner/plugins/system_info/` tree

## Recommended Migration Procedure

1. On a machine with Flutter installed, run:
   `flutter create --project-name flutter_showcase --platforms=windows .`
2. Review the generated `windows/` tree
3. Apply the permanent plugin wiring in `windows/runner/CMakeLists.txt`
4. Build once with `flutter build windows --release`
5. Commit the `windows/` directory
6. Simplify GitHub Actions by removing the Windows-specific generation and patch logic

## Why This Is Better

- FFI symbol export becomes visible in version control
- Windows build failures become reproducible outside CI
- Flutter template changes no longer silently break the patch script
- Reviewers can inspect runner-side native integration directly
